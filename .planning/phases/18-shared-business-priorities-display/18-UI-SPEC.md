## Typography

Same 4-role type scale as Phase 15/16/17. No new sizes, no new weights.

| Role | Size | Weight | Line Height | Phase 18 usage |
|------|------|--------|-------------|----------------|
| Body | 15px | 400 | 1.55 | Priority description paragraph, deliverable list-item text, sub-text under section header |
| Label / Metadata | 12px | 700 | 1.4 | Eyebrow (`SHARED FOCUS AREAS`, uppercase, letter-spacing 0.1em); priority eyebrow on Friday meeting (`BUSINESS PRIORITY 1 of 2`); `Show deliverables` / `Hide deliverables` toggle button (12px weight 700, no caps — matches `.role-read-more-btn` pattern from Phase 15) |
| Heading (section) | 20px | 700 | 1.3 | Section heading `Business Priorities` (h3) — matches Phase 15 `.personal-growth-section h3` and `.hub-section-toggle h3` at `src/index.css:1878–1882` |
| Heading (card) | 17px* | 700 | 1.3 | Priority title (h4) — see exception note below |
| Display | 28px | 700 | 1.2 | Not used in Phase 18 |

**\*Card title size — type-scale exception note:**

The CONTEXT.md D-07 description calls priority title an `h4`. The Phase 15 type scale declared exactly four sizes (12 / 15 / 20 / 28). Adding 17px would push the type scale to 5 sizes, breaking the Phase 15/16/17 contract. **Resolution (recommended default — autonomous mode):** priority title renders at **20px / weight 700 / line-height 1.3** using an `<h4>` semantic tag (markup) but the **same visual size as a section heading**. This keeps the type scale at exactly 4 sizes (12 / 15 / 20 / 28) and matches how Phase 15 `.weekly-choice-card h4` already rendered an `h4` at 15px — the visual size and the markup tag are independent in the Cardinal stylesheet. Each priority card ends up with a heading visually equal to the section heading; the eyebrow above each card disambiguates ("BUSINESS PRIORITY 1 of 2" inside the card vs no eyebrow on the section heading).

Source: Phase 15 UI-SPEC §Typography; `src/index.css` lines 47–53, 1878–1882, 2011–2014.

**Letter-spacing:** headings get the global `-0.02em` rule (line 48); eyebrows and uppercase metadata get `+0.1em`; body and toggle affordance use no letter-spacing.

**Italic:** none in Phase 18. (The `[TBD: replace via UPDATE before partner UAT]` placeholder is not styled — it is just part of the literal title/description text. Visible by default via standard text rendering.)

### `BusinessPrioritiesSection.jsx` (BIZ-02)

**File:** `src/components/BusinessPrioritiesSection.jsx`

**Prop contract:**
```jsx
<BusinessPrioritiesSection priorities={businessPriorities} />
```

- `priorities`: `Array<{ id: string, title: string, description: string, deliverables: string[] }>` — fetched by parent (`PartnerHub.jsx` or `AdminProfile.jsx`) via `fetchBusinessPriorities()` and passed in. Component does NOT fetch internally (matches PersonalGrowthSection prop pattern, CONTEXT D-08).
- No `partner` prop — content is shared (CONTEXT D-12).
- No callback props — read-only display (CONTEXT D-04, no write functions in v2.0).

**Loading + null fallback (CONTEXT D-08, D-09):**
- If `priorities` is `null` or `undefined` (parent still loading) → render `null` (matches PersonalGrowthSection's null-during-loading pattern; the parent `PartnerHub` already gates the entire hub on `loading` per existing convention).
- If `priorities` is `[]` (empty array — fetch succeeded but no rows) → render the section header + sub-text + a single muted message: `"No business priorities are configured yet."` (12px muted body, inside a `.business-priorities-empty` placeholder). This case should not occur in production (migration 011 always seeds 2 rows) but the component handles it defensively.
- If `priorities.length > 0` → render the populated state below.

**Layout (populated state):**
