## Component Inventory

New components and modified elements for Phase 10:

### 1. Meeting History Hub Card (PartnerHub.jsx ‚Äî modified)

Replace the existing "Meeting Summary" hub card. Use the identical `.hub-card` + `.hub-card-cta` pattern.

```
<Link to={`/meeting-history/${partner}`} className="hub-card">
  <h3>Meeting History</h3>
  <p>{description ‚Äî see Copywriting Contract}</p>
  <span className="hub-card-cta">Browse meetings ‚Üí</span>
</Link>
```

Visibility rule: Show only when `kpiLocked` is true (same gate as the current Meeting Summary card).

### 2. MeetingHistory.jsx ‚Äî New List Component

Layout: `.app-shell` > `.container` > `.screen.fade-in`

Structure within `.screen`:
1. Back navigation: `<Link className="btn-ghost">‚Üê Back to Hub</Link>` inside `.nav-row`
2. Eyebrow: `<div className="eyebrow">MEETING HISTORY</div>`
3. Screen header: `<h2>Past Meetings</h2>`
4. History list: `.meeting-history-list` (new class, mirrors `.scorecard-history-list` shape)
5. Each row: `.meeting-history-row` (new class, mirrors `.scorecard-history-row` shape ‚Äî clickable `<Link>`)

**History row anatomy** (one row per ended meeting):
- Week range: `formatWeekRange(week_of)` ‚Äî 14px weight 400, `--text`
- Type badge: inline colored dot + label (see Badge spec below)
- Ended date: `Ended {date}` ‚Äî 14px, `--muted`
- Right chevron: `‚Üí` in `--muted-2`

Full row text example: `Apr 7‚Äì13  ¬∑ Friday Review ¬∑  Ended Apr 11`

**Type badge spec** (new `.meeting-type-badge` class):
- Friday Review: red dot (`background: var(--red)`) + label "Friday Review" in `--muted`
- Monday Prep: blue dot (`background: var(--blue)`) + label "Monday Prep" in `--muted`
- Dot size: 8px √ó 8px, `border-radius: 50%`
- Layout: `display: inline-flex; align-items: center; gap: 6px; font-size: 13px`

Row hover: `border-color: var(--border-strong); background: var(--surface-2)` ‚Äî matches `.hub-card:hover` treatment.

### 3. MeetingSummary.jsx ‚Äî Modified (ID-based loading)

No visual changes to the summary detail view. Only the data-loading logic changes:
- Route becomes `/meeting-summary/:partner/:id`
- When `:id` present: call `fetchMeeting(id)` directly instead of scanning `fetchMeetings()`
- Back navigation: `‚Üê Back to History` linking to `/meeting-history/:partner` (instead of `‚Üê Back to Hub`)

The existing eyebrow, heading, stop blocks, and read-only treatment are unchanged.

## Interaction Contract

### MeetingHistory list navigation
- Entry: tap/click any `.meeting-history-row` ‚Üí navigate to `/meeting-summary/:partner/:id`
- Hover state: `translateY(-1px)` + `border-color: var(--border-strong)` (matches hub card hover)
- Active/press: `translateY(0)` (matches hub card active)
- Cursor: `pointer`

### MeetingSummary back navigation (from detail)
- "‚Üê Back to History" (`btn-ghost`) returns to `/meeting-history/:partner`
- This replaces the current "‚Üê Back to Hub" link only when navigated from the history list

### Loading state
- Both MeetingHistory and MeetingSummary: `return null` during loading (matches existing pattern in MeetingSummary and PartnerHub)
- No skeleton loader ‚Äî consistent with all existing components

### VALID_PARTNERS guard
- MeetingHistory.jsx: if `!VALID_PARTNERS.includes(partner)` ‚Üí `navigate('/', { replace: true })` ‚Äî same as MeetingSummary and PartnerHub
